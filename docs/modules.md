# windows-reactor module reference

This document maps every module in the `windows-reactor` crate, explains how
they relate to each other, and shows which parts are used by samples and tests.

## Public API surface

Everything flows through `lib.rs`, which re-exports from two private module
trees — `core` (platform-agnostic) and `winui` (WinUI 3 backend) — into a
single flat namespace. Users write `use windows_reactor::*` and never need to
name an interior module.

```text
lib.rs
├── pub use app::*            → App
├── pub use bootstrap::*      → bootstrap(), BootstrapHandle
├── pub use core::*           → (≈30 submodules, all re-exported flat)
├── pub use winui::backend    → WinUIBackend
├── pub use winui::dispatcher → WinUIDispatcher
├── pub use winui::hooks::*   → DispatcherTimer, Rendering, on_rendering
├── pub use winui::host::*    → ReactorHost, Backdrop, PresenterKind, …
├── pub use windows_core      → Error, Interface, Result
└── pub use windows_time      → DateTime, TimeSpan
```

## Module tree

### Top-level (private, re-exported)

| Module | Visibility | Purpose |
|--------|-----------|---------|
| `bindings` | private | Raw WinUI XAML bindings generated by `tool_bindings`. Never used directly. |
| `app` | private → `pub use` | `App` builder: window title, size, backdrop, presenter, and `run` / `render`. |
| `app_shim` | private | WinUI `Application` override plumbing (internal COM glue). |
| `bootstrap` | private → `pub use` | `bootstrap()` → `BootstrapHandle` for manual WinUI thread initialisation. |
| `diagnostics` | private | Panic hook, crash logs, `ExpectPanicGuard` (used by test harness). |
| `core` | private → `pub use` | Platform-agnostic reactor engine. See below. |
| `winui` | private → selective `pub use` | WinUI 3 backend, dispatcher, hooks, and host. See below. |

### `core` — the platform-agnostic engine

Every submodule is `pub` and glob-re-exported through `core/mod.rs`, so all
items land in the crate root namespace.

#### Rendering model

| Module | Key exports | Role |
|--------|------------|------|
| `component` | `Component<P>` trait | Stateless render unit; blanket-impl'd for `Fn(&P, &mut RenderCx) → Element`. |
| `component_element` | `ComponentElement`, `ComponentObject` trait | Type-erased wrapper that lets the reconciler store any `Component<P>`. |
| `render_context` | `RenderCx`, `SetState<T>`, `Updater<T>`, `Dispatch<A>`, `HookRef<T>`, `AsyncSetState<T>`, `UiMarshaller` | Mutable render context passed to every component; owns all hooks (see below). |
| `render_host` | `RenderHost`, `RenderStats`, `RenderCompleteInfo` | Drives the render loop: calls the root component, runs the reconciler, fires effects, tracks FPS/perf stats. |
| `reconciler` | `Reconciler<B>`, `RealizationQueue` | Diffs old vs. new element trees and issues backend mutations. Internal submodules: `child`, `diff_helpers`, `templated`, `widget_dispatch`, `wrappers`. |

#### Element tree

| Module | Key exports | Role |
|--------|------------|------|
| `element` | `Element` enum, `GroupElement`, `group()` | Sum type of every node kind: widget variants, `Component`, `Group`, `Custom`, `ErrorBoundary`, `TemplatedList`. |
| `element_ext` | `ElementExt` trait | Builder-style modifier methods (`.margin()`, `.background()`, `.with_key()`, animations, accessibility, …) implemented for every widget and for `Element`. |
| `into_elements` | `IntoElements` trait | Converts tuples, `Vec<Element>`, and arrays into child lists — enables `vstack((a, b, c))`. |
| `error_boundary` | `ErrorBoundaryElement`, `Fallback`, `error_boundary()` | Panic-catching wrapper that renders a fallback on descendant failure. |

#### Widgets

| Module | Key exports | Role |
|--------|------------|------|
| `widget` | `Widget` trait, `Children` enum | Adapter every built-in widget implements; exposes kind, bindings, children, header/pane slots, mount/unmount callbacks to the reconciler. |
| `widgets` (directory, 58 files) | One struct + factory fn per WinUI control | `button()`, `text_block()`, `grid()`, `vstack()`, `hstack()`, `slider()`, `combo_box()`, `navigation_view()`, `tab_view()`, `list_view()`, etc. Each file owns its struct, builder methods, `impl Widget`, and convenience constructor. |
| `generated_bindings` | `*_bindings()` fns | Auto-generated by `tool_reactor`; maps each widget struct to its `PropBindings` list. **Do not hand-edit.** |
| `prop_binding` | `Binding`, `PropBindings`, `Prop`, `PropValue`, `Event`, `EventHandler` | Declarative binding model: property values and event handler slots diffed by the reconciler. |

#### Hooks (on `RenderCx`)

All hooks are methods on `RenderCx` (in `render_context`). Summary:

| Hook | Signature | Purpose |
|------|-----------|---------|
| `use_state` | `(T) → (T, SetState<T>)` | Local component state with setter. |
| `use_reducer` | `(T) → (T, Updater<T>)` | Functional state updates via `f(prev) → next`. |
| `use_reducer_fn` | `(R, S) → (S, Dispatch<A>)` | Reducer with explicit action type. |
| `use_ref` | `(T) → HookRef<T>` | Persistent mutable ref that survives re-renders. |
| `use_memo` | `(deps, factory) → T` | Cached computation, recomputed when deps change. |
| `use_callback` | `(deps, f) → Callback<A>` | Stable callback identity, rebuilt when deps change. |
| `use_effect` | `(deps, f)` | Side-effect after render, re-runs when deps change. |
| `use_effect_with_cleanup` | `(deps, f) → cleanup` | Effect with teardown on re-run or unmount. |
| `use_context` | `(Context<T>) → T` | Read a value from an ancestor `Context` provider. |
| `use_async_state` | `(T) → (T, AsyncSetState<T>)` | Thread-safe state slot for async writes. |
| `use_color_scheme` | `() → ColorScheme` | Current light/dark theme. |
| `use_inner_size` | `() → WindowSize` | Current window inner dimensions. |
| `use_dpi` | `() → u32` | Current display DPI. |
| `use_ui_marshaller` | `() → UiMarshaller` | Thread-safe handle to enqueue work on the UI thread. |

#### Layout & styling

| Module | Key exports | Role |
|--------|------------|------|
| `geometry` | `Thickness`, `Color`, `Brush`, `GridLength`, `HorizontalAlignment`, `VerticalAlignment`, … | Layout primitives: margins, colors, brushes, grid sizing. |
| `modifiers` | `Modifiers`, `AttachedProps`, `AnimationModifiers`, `AccessibilityModifiers`, `Tooltip`, `PointerHandlers`, `GridPlacement` | Shared modifier bag carried by every widget element. |
| `animation` | `ScalarTransition`, `Vector3Transition`, `ImplicitTransitions`, `Vector3Axes` | Implicit animation descriptors applied via `ElementExt`. |
| `theme` | `ColorScheme`, `ThemeRef`, `BrushBinding`, `tokens` submodule | Theme-aware color references; `tokens::` contains named system-brush constants. |
| `resource` | `Resource<T>`, `use_resource`, `use_mutation` | Async data-loading lifecycle: `Idle → Loading → Ready(T) / Error`. |

#### Composition & extension

| Module | Key exports | Role |
|--------|------------|------|
| `context` | `Context<T>`, `ContextId`, `ContextStack`, `ContextProvision` | Ancestor-to-descendant value propagation (React-style context). |
| `custom` | `CustomElement` trait | Out-of-tree widget extension point; third-party leaf elements. |
| `templated_list` | `TemplatedListElement`, `TemplatedKind`, `SelectionMode`, `TemplatedListImpl` | Virtualised lists: `list_view`, `grid_view`, `flip_view` with item templates. |
| `rich_text` | `RichTextRun`, `RichTextInline`, `RichTextHyperlink`, `RichTextParagraph` | Structured rich-text model for `RichTextBlock`. |
| `keyboard` | `KeyboardAccelerator`, `KeyModifiers`, `KeyboardKey` | Keyboard shortcut bindings attached via modifiers. |

#### Plumbing

| Module | Key exports | Role |
|--------|------------|------|
| `backend` | `Backend` trait, `ControlId`, `ControlKind` | Abstract backend interface; every operation the reconciler can request. |
| `dispatcher` | `Dispatcher` trait, `SendDispatcher` trait, `DispatcherQueuePriority` | Thread-dispatching abstraction. |
| `callback` | `Callback<T>`, `IntoCallback<T>` | `Rc`-pointer-equal function wrapper; structural identity for the diffing engine. |
| `rc_fn` | `impl_rc_fn_wrapper!` macro | Generates `Rc<dyn Fn>`-backed newtypes with `Clone`, `PartialEq` (by pointer), `Debug`. |

### `winui` — the WinUI 3 backend

| Module | Key exports | Role |
|--------|------------|------|
| `backend` | `WinUIBackend` | Implements the `Backend` trait against live XAML controls. Submodules: `convert` (Rust↔XAML type conversions), `diag` (missing-feature warnings), `generated_set_prop` / `generated_attach_event` (codegen'd property and event wiring). |
| `dispatcher` | `WinUIDispatcher` | `Dispatcher` + `SendDispatcher` over `DispatcherQueue`. |
| `hooks` | `DispatcherTimer`, `Rendering`, `on_rendering()` | WinUI-specific timing hooks: periodic timer and composition-frame callback. |
| `host` | `ReactorHost`, `Backdrop`, `PresenterKind`, `RequestedTheme`, `set_requested_theme()`, `set_titlebar_height()` | Window creation, sizing, theme, titlebar, fullscreen, and backdrop configuration. |

### Import patterns

Most consumers use `use windows_reactor::*` — the flat re-export design means
a single glob import gives access to every widget constructor, hook, layout
type, and trait. Only the selftest harness cherry-picks individual items for
clarity (`Component`, `RenderHost`, `WinUIBackend`, `WinUIDispatcher`).

## Roles vs. module structure

The 27 submodules inside `core` serve only ~6 conceptual roles. The current
one-file-per-concept split creates navigational overhead without providing
meaningful encapsulation (everything is `pub` and glob-re-exported anyway).
Here's how the modules map to their actual *purpose*:

### Conceptual roles

| Role | Current modules | What it does |
|------|----------------|--------------|
| **Render engine** | `component`, `component_element`, `render_context`, `render_host`, `reconciler` (+ 5 internal submodules) | The core loop: components produce elements, hooks track state, the reconciler diffs and patches. |
| **Element tree** | `element`, `element_ext`, `into_elements`, `error_boundary`, `custom` | Definition of the `Element` enum, modifier chaining, tuple→children conversion, panic recovery, and the extension-point trait. |
| **Widgets** | `widget`, `widgets` (57 files), `prop_binding`, `generated_bindings`, `templated_list`, `rich_text` | Concrete control structs, the `Widget` trait, and the binding/diffing vocabulary. |
| **Styling & layout** | `geometry`, `modifiers`, `animation`, `theme` | Types for margins, colors, brushes, grid lengths, implicit transitions, theme tokens — all carried inside `Modifiers`. |
| **Interaction** | `keyboard`, `callback`, `rc_fn`, `resource`, `context` | Input handling, stable function wrappers, async data, and ancestor-value propagation. |
| **Backend contract** | `backend`, `dispatcher` | Trait interfaces the platform layer must implement. |

### Observations

1. **Splitting by type rather than purpose.** `component` and
   `component_element` are the same concept (a renderable unit) in two files.
   `element`, `element_ext`, and `into_elements` are three facets of one type.
   `callback` and `rc_fn` exist only because the macro lives separately from
   its first user. None of these splits help a reader understand the system.

2. **`render_context` is doing triple duty.** It defines all hooks (`use_state`,
   `use_effect`, …), the `RenderCx` struct itself, and several supporting
   newtypes (`SetState`, `Updater`, `Dispatch`, `HookRef`). It's the largest
   file in the crate. Meanwhile `render_host` is a thin driver that could be
   folded in or kept as a sibling — the two are always read together.

3. **Styling is scattered.** `geometry` defines `Thickness`, `Color`, `Brush`,
   `GridLength`, etc.; `modifiers` defines the bag that *holds* those values;
   `animation` defines transition descriptors that live *inside* that bag;
   `theme` defines brush references that resolve *into* those values. A single
   `style` module would keep the full story in one place.

4. **`prop_binding` and `generated_bindings` are widget internals.** They exist
   only to serve the reconciler's widget-diffing path. They could live inside
   the `widgets` directory rather than as peer modules to `element` and
   `component`.

5. **The `widgets/` directory is correctly granular.** One file per control is
   the right call — controls are independent and numerous. This is the one area
   where high file count helps (you find a widget by name).

### Current structure (consolidated)

The consolidation has been applied. The `core/` directory now contains:

```text
core/
├── engine.rs              Component, ComponentElement, RenderCx (hooks),
│                          RenderHost, RenderStats
├── element.rs             Element enum, GroupElement, ElementExt, IntoElements,
│                          ErrorBoundary, CustomElement
├── style.rs               Thickness, Color, Brush, GridLength, Modifiers,
│                          ImplicitTransitions, ThemeRef, BrushBinding, tokens
├── backend.rs             Backend trait, ControlId, ControlKind, Dispatcher,
│                          SendDispatcher, UiMarshaller
├── widget.rs              Widget trait, PropBindings, Binding, Prop, PropValue,
│                          Event, EventHandler, TemplatedList, RichText
├── interaction.rs         Callback, RcFn macro, KeyboardAccelerator, Context,
│                          Resource
├── reconciler.rs          Reconciler<B> + internal submodules (child,
│                          diff_helpers, templated, widget_dispatch, wrappers)
├── generated_bindings.rs  Auto-generated prop-binding functions (tool_reactor)
├── widgets/               57 per-control files, one struct+factory each
└── mod.rs                 9 module declarations + glob re-exports
```

This reduces the original 27 modules to 9 (7 hand-written + reconciler +
generated). Each file answers one question: "how does the render loop work?",
"what is an element?", "how do I style things?", etc. The flat re-export at
the crate root means zero API breakage — only internal `use super::*` paths
